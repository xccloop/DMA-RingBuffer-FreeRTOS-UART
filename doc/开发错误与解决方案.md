## 第1轮：318 errors — `__IO`, `uint32_t`, `HAL_StatusTypeDef` 等未定义

### 症状
```
error: #20: identifier "__IO" is undefined
error: #757: variable "uint32_t" is not a type name
error: #20: identifier "HAL_StatusTypeDef" is undefined
error: #20: identifier "RCC" is undefined
error: #20: identifier "GPIO_InitTypeDef" is undefined
```

### 根因（3个同时存在）

**根因1**：`System/` 缺少 `stm32f103xe.h` 设备头文件。此文件定义所有外设寄存器地址、中断号、GPIO/RCC/USART 结构体。`stm32f1xx.h` 会 `#include "stm32f103xe.h"`，文件不存在时编译中断。

获取方式：
```powershell
git clone --depth 1 https://github.com/STMicroelectronics/cmsis_device_f1.git
# 复制 Include/stm32f103xe.h → System/
```

**根因2**：`HAL/inc/stm32f1xx_hal.h` 缺少 `#include "stm32f1xx_hal_def.h"`。CMSIS 类型（`__IO`、`uint32_t`）的定义链路是：
```
stm32f1xx_hal.h → stm32f1xx_hal_def.h → stm32f1xx.h → core_cm3.h → __IO, uint32_t
```
缺少 `stm32f1xx_hal_def.h` 的 include 导致整个类型系统不可见。

修复：在 `stm32f1xx_hal.h` 第 29 行之后加一行：
```c
#include "stm32f1xx_hal_conf.h"
#include "stm32f1xx_hal_def.h"   // ← 新增
```

**根因3**：`Framework/uart_api.c` 直接访问不透明结构体 `drv->rx_task`，但 `uart_dma_handle_t` 的完整定义在 `uart_dma.c` 内部。

修复：在 `uart_dma.h` 新增 setter 函数：
```c
void uart_dma_set_rx_task(uart_dma_handle_t *h, void *task);
```
在 `uart_api.c` 中把 `c->drv->rx_task = ...` 替换为 `uart_dma_set_rx_task(c->drv, ...)`。

## 第2轮：12 errors — `Legacy/stm32_hal_legacy.h` 找不到

### 症状
```
error: #5: cannot open source input file "Legacy/stm32_hal_legacy.h"
  #include "Legacy/stm32_hal_legacy.h"
```

### 根因
`stm32f1xx_hal_def.h` 第 30 行：`#include "Legacy/stm32_hal_legacy.h"`。文件实际在 `HAL/inc/stm32_hal_legacy.h`，不在 `Legacy/` 子目录。

### 解决方案
```powershell
mkdir HAL\inc\Legacy
move HAL\inc\stm32_hal_legacy.h HAL\inc\Legacy\
```

## 第4轮：161 errors — HAL 模块类型大面积未定义

### 症状
```
error: #20: identifier "GPIO_InitTypeDef" is undefined
error: #20: identifier "UART_HandleTypeDef" is undefined
error: #20: identifier "DMA_HandleTypeDef" is undefined
error: #20: identifier "FLASH_ProcessTypeDef" is undefined
```

### 根因
每个 HAL .c 文件只 include 了 `stm32f1xx_hal.h` 一个头文件，依赖它级联引入所有模块头文件。但是 `stm32f1xx_hal.h` 末尾没有包含模块头文件。

HAL 设计模式：
```
stm32f1xx_hal.c 只 include "stm32f1xx_hal.h"
  → stm32f1xx_hal.h 应该根据 stm32f1xx_hal_conf.h 中的宏，
    条件 include 每个启用的模块头文件
```

### 解决方案
在 `stm32f1xx_hal.h` 末尾（`#endif` 之前）添加条件 include：
```c
#ifdef HAL_GPIO_MODULE_ENABLED
#include "stm32f1xx_hal_gpio.h"
#endif
#ifdef HAL_DMA_MODULE_ENABLED
#include "stm32f1xx_hal_dma.h"
#endif
#ifdef HAL_UART_MODULE_ENABLED
#include "stm32f1xx_hal_uart.h"
#endif
#ifdef HAL_RCC_MODULE_ENABLED
#include "stm32f1xx_hal_rcc.h"
#endif
#ifdef HAL_CORTEX_MODULE_ENABLED
#include "stm32f1xx_hal_cortex.h"
#endif
#ifdef HAL_FLASH_MODULE_ENABLED
#include "stm32f1xx_hal_flash.h"
#endif
#ifdef HAL_PWR_MODULE_ENABLED
#include "stm32f1xx_hal_pwr.h"
#endif
```

## 第5轮：5 errors — RCC 超时常量未定义

### 症状
```
error: #20: identifier "HSE_TIMEOUT_VALUE" is undefined
error: #20: identifier "RCC_LSE_TIMEOUT_VALUE" is undefined
```

### 根因
`stm32f1xx_hal_rcc.h` 中定义：
```c
#define HSE_TIMEOUT_VALUE    HSE_STARTUP_TIMEOUT
#define RCC_LSE_TIMEOUT_VALUE  LSE_STARTUP_TIMEOUT
```
但 `HSE_STARTUP_TIMEOUT` 和 `LSE_STARTUP_TIMEOUT` 需要用户在 `stm32f1xx_hal_conf.h` 中定义。

### 解决方案
在 `User/stm32f1xx_hal_conf.h` 中添加：
```c
#define HSE_STARTUP_TIMEOUT  ((uint32_t)100)    /* 100 ms */
#define LSE_STARTUP_TIMEOUT  ((uint32_t)5000)   /* 5000 ms */
```

## 第6轮：1 error (链接) — `SysTick_Handler` 重复定义

### 症状
```
Error: L6200E: Symbol SysTick_Handler multiply defined (by port.o and stm32f1xx_it.o).
```

### 根因
- FreeRTOS 的 `port.c` 定义了 `xPortSysTickHandler`
- `FreeRTOSConfig.h` 中有 `#define xPortSysTickHandler SysTick_Handler`，这使编译器把它重命名为 `SysTick_Handler`
- `stm32f1xx_it.c` 中也定义了 `SysTick_Handler`（调用 `HAL_IncTick()` + `xPortSysTickHandler()`）
- 链接时两个 .o 文件都有 `SysTick_Handler` 符号 → 冲突

### 解决方案
删除 `FreeRTOSConfig.h` 中的宏映射：
```c
// 删除这行:
// #define xPortSysTickHandler   SysTick_Handler

// 改为注释:
/* xPortSysTickHandler is called from SysTick_Handler in stm32f1xx_it.c */
```
这样 `port.c` 中的函数保持原名 `xPortSysTickHandler`，由 `stm32f1xx_it.c` 中的 `SysTick_Handler` 调用，不再冲突。

## 第7轮：4 errors (链接) — 未定义符号

### 症状
```
Error: L6218E: Undefined symbol vTaskDelay (referred from main.o).
Error: L6218E: Undefined symbol xTaskDelayUntil (referred from main.o).
Error: L6218E: Undefined symbol FLASH_PageErase (referred from stm32f1xx_hal_flash.o).
Error: L6218E: Undefined symbol __CLZ (referred from stm32f1xx_hal_pwr.o).
```

### 根因

**vTaskDelay / xTaskDelayUntil**：
FreeRTOS 默认不编译这些函数。`FreeRTOS.h` 中有：
```c
#ifndef INCLUDE_vTaskDelay
    #define INCLUDE_vTaskDelay    0   // 默认关闭!
#endif
#ifndef INCLUDE_xTaskDelayUntil
    #define INCLUDE_xTaskDelayUntil    0   // 默认关闭!
#endif
```
需要在 `FreeRTOSConfig.h` 中显式启用。

**FLASH_PageErase**：HAL flash 驱动声明为 `extern`，期望外部提供实现。这是 CMSIS 设备层函数。

**__CLZ**：CMSIS 的 Count Leading Zeros 函数。`core_cm3.h` 中为 `static inline`，但编译器可能无法内联，需要外部定义。

### 解决方案

**步骤1**：在 `FreeRTOSConfig.h` 中启用 API：
```c
#define INCLUDE_vTaskDelay               1
#define INCLUDE_xTaskDelayUntil          1
```

**步骤2**：新建 `User/stubs.c` 提供缺失函数：
```c
#include "stm32f1xx_hal.h"

void FLASH_PageErase(uint32_t PageAddress)
{
    while (FLASH->SR & FLASH_SR_BSY);
    if (FLASH->CR & FLASH_CR_LOCK) {
        FLASH->KEYR = 0x45670123;
        FLASH->KEYR = 0xCDEF89AB;
    }
    FLASH->CR |= FLASH_CR_PER;
    FLASH->AR   = PageAddress;
    FLASH->CR  |= FLASH_CR_STRT;
    while (FLASH->SR & FLASH_SR_BSY);
    if (FLASH->SR & FLASH_SR_EOP) FLASH->SR = FLASH_SR_EOP;
    FLASH->CR &= ~FLASH_CR_PER;
    FLASH->CR |= FLASH_CR_LOCK;
}

uint8_t __CLZ(uint32_t data)
{
    uint8_t cnt = 0;
    if (data == 0) return 32;
    while ((data & 0x80000000) == 0) { data <<= 1; cnt++; }
    return cnt;
}
```

**步骤3**：在 Keil 工程中把 `User/stubs.c` 加入 User 组。

## 第8轮：1 error (链接) — API 名称不匹配

### 症状
```
Error: L6218E: Undefined symbol xTaskDelay (referred from main.o).
```

### 根因
本项目的 FreeRTOS 源码版本中只有 `vTaskDelay`（旧名称），没有 `xTaskDelay`（新名称）。之前曾把 main.c 中的 `vTaskDelay` 改成了 `xTaskDelay`，但 tasks.c 只提供了 `vTaskDelay`。

### 解决方案
`main.c` 使用 `vTaskDelay`（源码中存在的函数名），使用 `xTaskDelayUntil`（源码中存在的函数名）。
```c
// ✅ 正确：
vTaskDelay(pdMS_TO_TICKS(100));
xTaskDelayUntil(&t, pdMS_TO_TICKS(500));
```

## 排查方法论总结

遇到 HAL 库编译错误时，按以下顺序排查：

1. **`identifier "__XX" is undefined`** → CMSIS 类型缺失 → 检查 `stm32f1xx.h` → `stm32f103xe.h` → `core_cm3.h` include 链
2. **`cannot open source input file "XXX.h"`** → 头文件缺失或路径不对 → 检查文件是否存在、include path 是否包含
3. **`identifier "XXX_HandleTypeDef" is undefined`** (大量出现) → 模块头文件未级联 → `stm32f1xx_hal.h` 是否包含了对应模块头
4. **`identifier "XXX_TIMEOUT_VALUE" is undefined`** → 用户配置常量缺失 → `stm32f1xx_hal_conf.h` 是否定义
5. **`Symbol XXX multiply defined`** (链接) → 重复定义 → 检查是否有宏重命名 + 同名函数
6. **`Undefined symbol XXX`** (链接) → 函数未编译或未链接 → 检查 `#if` 开关、extern 声明是否有对应实现
