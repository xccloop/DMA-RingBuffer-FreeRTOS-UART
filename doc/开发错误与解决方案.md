# DMA + RingBuffer + FreeRTOS UART 通信框架 —— 开发错误与解决方案

本文档记录了在开发此框架过程中遇到的典型错误、根因分析及解决方案。每个错误都附有症状描述和修复代码，可作为调试参考手册。

---

## 错误 1：IDLE 中断中 NDTR 读取时序错误导致丢字节

### 症状
- 接收变长数据时，帧末尾 1~2 个字节偶尔丢失
- 高波特率（> 460800bps）下问题频发
- 低频（9600bps）下几乎不出现

### 根因
IDLE 中断中先读了 NDTR（DMA 剩余计数），然后才通过读 SR+DR 清除 IDLE 标志：

```c
// ❌ 错误顺序
void USART_IRQHandler(void)
{
    uint32_t ndtr = DMA1_Channel5->CNDTR;  // 先读 NDTR ← 问题!
    uint32_t sr = USART1->SR;
    uint32_t dr = USART1->DR;              // 后清 IDLE
    ...
}
```

在读取 NDTR 和清除 IDLE 之间，DMA 可能又传输了 1 个字节。此时 NDTR 减少了 1，但计算时用的是旧的 NDTR，导致少计 1 字节。更严重的是，如果恰好在此窗口内完成了下一个字节的接收，这个字节会被归入下一帧。

### 解决方案
**先清除 IDLE，再读 NDTR**：

```c
// ✅ 正确顺序
void USART_IRQHandler(void)
{
    if (USART1->SR & USART_SR_IDLE) {
        uint32_t sr = USART1->SR;           // 1. 读 SR
        uint32_t dr = USART1->DR;           // 2. 读 DR 清除 IDLE ← 必须先做
        uint32_t ndtr = DMA1_Channel5->CNDTR; // 3. 再读 NDTR
        // 此时 NDTR 快照是 IDLE 清除后的稳定值
        ...
    }
}
```

**原理**：读 SR 后读 DR 会硬件清除 IDLE 标志（这是 STM32 参考手册规定的序列）。在清除 IDLE 之后，即使 DMA 继续传输，计算出的 received 也是"到当前帧结束"的正确值。

## 错误 2：DMA TX Normal 模式未等待上一次完成就重新配置

### 症状
- 连续快速调用 `uart_send()` 时，前一次发送被截断
- 数据中间出现空洞（丢失几字节到几十字节）
- DMA 错误标志（TEIF）偶尔置位

### 根因
DMA_Channel 的 CCR 寄存器 EN 位在传输进行中时不能被直接修改。如果在 EN=1 时写 CMAR/CPAR/CNDTR，结果是未定义的（STM32F1 上通常被忽略，导致新配置不生效）。

```c
// ❌ 错误：未检查 DMA 是否忙
void dma_tx_kick(void)
{
    DMA1_Channel4->CMAR = new_data;   // 如果 DMA 正在跑, 这行可能无效!
    DMA1_Channel4->CNDTR = new_len;
    DMA1_Channel4->CCR |= DMA_CCR_EN;
}
```

### 解决方案
**配置 DMA 前必须先清零 EN 位**：

```c
// ✅ 正确：先停止，再配置
void dma_tx_kick(void)
{
    DMA1_Channel4->CCR &= ~DMA_CCR_EN;    // 1. 先关 EN
    // 等待硬件确认 EN=0（通常 1~2 个 AHB 周期）
    while (DMA1_Channel4->CCR & DMA_CCR_EN);
    
    DMA1_Channel4->CMAR = new_data;       // 2. 安全配置
    DMA1_Channel4->CNDTR = new_len;
    DMA1_Channel4->CCR |= DMA_CCR_EN;     // 3. 启动
}
```

框架在 `uart_port_dma_tx_start()` 中已正确处理此问题。

## 错误 3：RingBuffer head/tail 使用 32 位自增导致取模误差

### 症状
- 长时间运行后（数小时至数天），接收数据出现字节错位
- RingBuffer 报告 `available` 和实际可读数据量不符

### 根因
初版实现使用 `head % capacity` 计算数组索引：

```c
// ❌ 风险：32 位取模在 capacity 非 2 的幂时用硬件除法，有微小性能开销
// 更严重的是：如果 capacity 意外传入非 2 的幂值...
uint32_t index = head % capacity;  // 龟速且不安全
```

但这不是主要问题。真正的问题是：如果 `capacity` 被意外设为非 2 的幂值（如 500），则 `head - tail` 可能不等于实际数据量。因为 `head % 500` 在 head 溢出 32 位（约 42.9 亿字节后）时，head 回绕到 0，但 tail 还在 42.9 亿附近，导致 `head - tail` 出现巨大的负值（转为 uint32_t 后是一个巨大的正数）。

虽然 32 位溢出的理论时间很长（6Mbps 下 ~95 年），但在测试中使用小容量 buffer + 高速发送可以加速触发。

### 解决方案
**capacity 必须是 2 的幂，并在初始化时断言检查**：

```c
// ✅ ringbuffer.c 实现
void rb_init(ringbuffer_t *rb, uint8_t *buf, uint32_t capacity)
{
    // 拒绝非 2 的幂
    if ((capacity & (capacity - 1)) != 0) {
        return;  // 初始化失败，buffer 指针保持 NULL
    }
    rb->capacity = capacity;
    rb->mask = capacity - 1;
    ...
}

// 取模用位与，零开销
uint32_t index = head & rb->mask;
```

## 错误 4：从 ISR 中调用 FreeRTOS API 前未检查优先级

### 症状
- 偶发性 HardFault
- 在 DMA TC 中断中调用 `xSemaphoreGiveFromISR` 时崩溃
- 问题在轻度负载时不出现，CPU 负载增加后出现

### 根因
FreeRTOS 要求在 ISR 中调用的 API（如 `xSemaphoreGiveFromISR`、`vTaskNotifyGiveFromISR`）必须在逻辑优先级 **低于** `configMAX_SYSCALL_INTERRUPT_PRIORITY` 的中断中执行。

如果 DMA 中断优先级被配置为 0（最高优先级，即逻辑优先级 > `configMAX_SYSCALL_INTERRUPT_PRIORITY`），则在 ISR 中调用 `xQueueSendFromISR` 等函数会触发 `configASSERT` 失败（或直接 HardFault）。

默认 FreeRTOS 配置中 `configMAX_SYSCALL_INTERRUPT_PRIORITY` 通常为 5（取决于 `configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY`），所以 DMA 中断的 NVIC 优先级必须 ≥ 5（数值上）。

### 解决方案
**正确配置中断优先级**：

```c
// ✅ 正确优先级设置
// FreeRTOS 可管理的优先级范围: 5 ~ 15（数值）
// DMA 中断优先级应为 5~7（留有余量）

NVIC_SetPriority(DMA1_Channel4_IRQn, 7);  // DMA TX: 优先级 7
NVIC_SetPriority(DMA1_Channel5_IRQn, 7);  // DMA RX: 优先级 7
NVIC_SetPriority(USART1_IRQn,        6);  // USART: 优先级 6 （高于 DMA 以便及时清 IDLE）

// configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY = 5
// 优先级 5 以上的 ISR 可以调用 FreeRTOS FromISR API
```

**注意**：STM32 NVIC 只使用 4 位优先级（STM32F103 上），所以优先级值的有效范围是 0-15。数值越小优先级越高（Cortex-M 标准）。

## 错误 5：环形缓冲区写满后的静默覆盖

### 症状
- 接收任务消费速度跟不上时，旧数据被静默覆盖
- 应用层完全不知道数据丢失

### 根因
初版 DMA RX ISR 中直接更新 head，未检查是否会越过 tail：

```c
// ❌ 未检查溢出
rb->head += received;  // 直接加，head 越过 tail 意味着覆盖未读数据
```

### 解决方案
在 `rb_advance_head` 中添加溢出检测和处理：

```c
// ✅ 带溢出检测
void rb_advance_head(ringbuffer_t *rb, uint32_t len)
{
    uint32_t free = rb_free_space(rb);  // capacity - (head - tail)
    if (len > free) {
        // 溢出：丢弃最旧的数据以腾出空间
        uint32_t overflow = len - free;
        rb->tail += overflow;  // 旧数据被跳过
        // 可在此处设置溢出标志通知应用层
    }
    rb->head += len;
}
```

另外，在应用层 API 中暴露了 `uart_get_errors()` 接口，返回包含 ORE 标志的错误寄存器，应用层可定期检查。

## 错误 6：DMA TX 分段传输时 snap_head 被新数据覆盖

### 症状
- 在发送一段大数据期间，应用又写入了新数据
- 第二段 DMA 传输的数据中包含部分旧数据和部分新数据
- 接收端看到数据交错

### 根因
TX DMA 分段传输流程中，第一次 `dma_tx_kick` 时对 head 做了快照（`tx_kick_head = head`）。但在第一段 DMA 传输期间，应用可能又调用了 `uart_send`，导致 head 再次前进。如果在 TC 中断中，用"当前 head"而不是"快照 head"来计算第二段传输长度，就会把新写入的数据也发出去（但新数据前面的部分已被第一段 DMA 覆盖）。

### 解决方案
**在整个分段传输过程中使用同一个快照 head**：

```c
static void dma_tx_kick(uart_dma_handle_t *h)
{
    uint32_t head = rb_get_head(&h->tx_rb);
    uint32_t tail = rb_get_tail(&h->tx_rb);
    
    h->tx_kick_head = head;  // ← 关键：快照

    if (tail < head) {
        // 无回绕：一段完成
        uint32_t seg_len = head - tail;
        uart_port_dma_tx_start(h->port, &h->tx_rb.buffer[tail & mask], seg_len);
    } else {
        // 有回绕：第一段 [tail, capacity)
        uint32_t seg1_len = h->tx_rb.capacity - (tail & mask);
        h->tx_chain_pending = ((head & mask) > 0);  // 是否需要第二段
        uart_port_dma_tx_start(h->port, &h->tx_rb.buffer[tail & mask], seg1_len);
    }
}
```

TC 中断中：
```c
static void dma_tx_tc_isr(uint8_t port)
{
    if (h->tx_chain_pending) {
        // 第二段只发 [0, tx_kick_head) — 不包含 kick 之后新写入的数据
        uint32_t seg2_len = h->tx_kick_head & h->tx_rb.mask;
        uart_port_dma_tx_start(h->port, &h->tx_rb.buffer[0], seg2_len);
        h->tx_chain_pending = false;
    } else {
        // 一段完成：推进 tail 到 snap_head
        h->tx_rb.tail = h->tx_kick_head;  // ← 使用快照值
        // 然后检查是否有新数据（当前 head != 快照 head）
        if (rb_get_head(&h->tx_rb) != h->tx_kick_head) {
            dma_tx_kick(h);  // 重新 kick
        }
    }
}
```

这样保证：在"第一段 TC → 第二段启动"和"全部完成 → 检查新数据"之间，新到达的数据不会混入当前传输。

## 错误 7：FreeRTOS Mutex 死锁（自嵌套）

### 症状
- 在 `uart_printf()` 的实现中调用 `uart_send()`，导致同一 Mutex 被同一任务获取两次
- 任务永久阻塞

### 根因
```c
// ❌ uart_printf 内部调用 uart_send
int32_t uart_printf(uint8_t port, const char *fmt, ...)
{
    char buf[256];
    vsnprintf(buf, ...);

    // 错误：此时如果已经持有 tx_mutex（外部调了 uart_send 后调 uart_printf），死锁！
    return uart_send(port, buf, len, timeout);
}
```

FreeRTOS Mutex 不支持递归（同一任务不能重复获取）。

### 解决方案
**将 uart_printf 的核心逻辑复用 uart_send 的底层调用，而非 uart_send 本身**。在本框架中，`uart_printf` 调用 `uart_send` 之前已经先获取了 mutex。为了避免递归，`uart_printf` 的实现中直接调用 `uart_send`（`uart_send` 会自己获取 mutex）。

实际上，本框架的 `uart_printf` 调用的是 `uart_send`（公共 API），这本身是安全的——因为 `uart_send` 内部首先做的就是 `xSemaphoreTake(mutex)`。如果当前任务已经持有该 mutex（从外部），则会在 Take 处阻塞。所以在使用上需要注意：

**使用准则**：不要在 `uart_send()` 的参数中调用 `uart_printf()`，反之亦然。如果需要在持有 mutex 的情况下发送数据，使用底层 `uart_dma_send` 直接操作。

在框架的最终实现中，`uart_printf` 内部有自己独立的 mutex 获取流程，不会与其他 `uart_send` 调用互相死锁，因为它们先释放再获取。

## 错误 8：DMA RX 循环模式缓冲区地址对齐

### 症状
- DMA 配置无误但接收数据出现随机错误
- 某些字节被 0x00 或 0xFF 替换
- 问题只在开优化（-O2）时出现

### 根因
DMA 循环模式要求缓冲区地址满足 DMA 的对齐要求。虽然字节传输（8-bit）不强制要求对齐，但某些 STM32F1 的硬件版本中，即使 8 位宽度的 DMA 传输也建议缓冲区 4 字节对齐。

如果 RingBuffer 的 `buffer` 字段通过 `pvPortMalloc` 分配，FreeRTOS 的 heap_4 分配器返回 8 字节对齐的地址，所以通常不是问题。但如果用静态数组：

```c
static uint8_t rx_buf[1024];  // 编译器可能分配在非对齐地址
```

### 解决方案
**使用对齐属性声明静态缓冲区**：

```c
static uint8_t rx_buf[1024] __attribute__((aligned(4)));
```

或使用 `pvPortMalloc` 动态分配（已保证对齐）。

## 错误 9：UART 初始化后未清空 SR 中的残留标志

### 症状
- `uart_init` 后立即触发 IDLE 中断（无实际数据）
- 或首次接收时多出 0x00 字节

### 根因
上电后 USART_SR 中可能有随机值（或上次复位残留的标志位）。特别是：
- TC（发送完成）标志在复位后默认为 1
- 如果未清除这些标志就使能中断，会立即触发一次虚假中断

### 解决方案
**在配置 UART 之前，先读 SR 再读 DR 清除所有标志**：

```c
void uart_port_init(uint8_t port, uint32_t baudrate)
{
    // 清除所有可能的残留标志
    volatile uint32_t sr = usart->SR;
    volatile uint32_t dr = usart->DR;
    (void)sr;
    (void)dr;
    
    // 然后配置 CR1/CR2/CR3...
    // 最后使能 UE
}
```

## 错误 10：在 DMA 中断和 UART 中断间共享变量的可见性问题

### 症状
- DMA TC 中断中更新的 `tail`，在 UART IDLE 中断中读取到旧值
- 编译器优化导致变量缓存在寄存器中
- 仅在开启 -O2 或 -Os 优化时出现

### 根因
Cortex-M3 上，不同优先级中断之间没有数据竞争问题（高优先级中断打断低优先级中断，返回后低优先级的 CPU 寄存器被恢复）。但**相同优先级**的中断不会嵌套，所以同一个优先级的中断之间是安全的。

然而，编译器可能将 `head`/`tail` 缓存在寄存器中（在同一个执行上下文中连续访问）。如果 DMA ISR（优先级 7）和 UART IDLE ISR（优先级 6）可以互相抢占，则可能出现：
1. UART ISR 读取 `head` → 缓存在寄存器 R5
2. DMA ISR 抢占，修改 `head`（内存）
3. UART ISR 恢复，继续使用 R5 中的旧 `head` 值

### 解决方案
**使用 `volatile` 修饰 ringbuffer 的 head 和 tail**：

```c
typedef struct {
    volatile uint32_t head;    // ← volatile 阻止编译器缓存到寄存器
    volatile uint32_t tail;    // ← 每次访问都从内存重新读取
    ...
} ringbuffer_t;
```

加上 **DMA ISR 优先级低于 UART ISR 优先级**（DMA=7, UART=6），确保 IDLE ISR 不会被 DMA ISR 中断（IDLE 优先级更低、数值更高意味着可以被 DMA 中断），但 DMA ISR 执行时间 < 1μs，影响可忽略。

## 总结：调试 DMA+UART 问题的系统方法

1. **先查硬件**：TX/RX 引脚配置（AF_PP / INPUT）、波特率匹配、外部晶振频率
2. **再查寄存器**：用调试器查看 NDTR、SR、CCR 寄存器的实时值，确认 DMA 是否在运行
3. **最小化复现**：关闭所有非必要中断，用最简代码（一个 echo）复现问题
4. **逻辑分析仪**：在 TX/RX 引脚上接逻辑分析仪，观察波形是否与预期一致
5. **添加诊断日志**：在关键路径（ISR 入口/出口、head/tail 更新前后）输出调试信息
6. **回归测试**：修复后运行测试用例（回绕、溢出、并发写入）验证修复
